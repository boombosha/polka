# Polka — стоковые изображения по ключевому слову

Цель секции: по ключевому слову получать изображение из бесплатного фотостока по API, ресайзить до нужного размера и кешировать результат, чтобы повторные запросы не ходили во внешний API.

## 1) Бесплатные фотостоки с API

### Рекомендуемые кандидаты

1. **Pexels API**
   - Плюсы: простая интеграция, бесплатный API-ключ, хорошие метаданные, удобный поиск.
   - Минусы: лимиты запросов, требования атрибуции и правил использования.
   - Подходит как основной провайдер.

2. **Pixabay API**
   - Плюсы: бесплатный API, стабильный поиск, много изображений.
   - Минусы: качество и выдача по некоторым темам может быть менее предсказуема.
   - Хорош как fallback-провайдер.

3. **Unsplash API**
   - Плюсы: очень качественный контент, развитая экосистема.
   - Минусы: более строгие правила использования API и отображения источника.
   - Подходит, если готовы строго соблюдать гайдлайны и квоты.

### Практическая рекомендация для MVP

- Основной провайдер: **Pexels**
- Fallback: **Pixabay**
- Unsplash подключать вторым этапом, если потребуется качество/разнообразие и готовы к stricter compliance.

## 2) Предлагаемый API-контракт в Polka

Префикс фичи: **`/stock`**

Пример:

- `GET /stock/:width/:height/:query`
- Query-параметры:
  - `fit` = `cover|contain|inside` (по умолчанию `cover`)
  - `provider` = `auto|pexels|pixabay` (по умолчанию `auto`)
  - `format` = `webp|jpeg|png` (по умолчанию `webp`)
  - `q` = качество (например `1..100`, по умолчанию `82`)
  - `seed` = фиксатор детерминизма (опционально)

Ответ:
- `Content-Type: image/webp|image/jpeg|image/png`
- Заголовки кеширования (см. стратегию ниже)

## 3) Стратегия кеширования (рекомендуемая)

### Многоуровневый кеш

1. **L1 (in-memory, короткий)**
   - Быстрый ответ для горячих запросов.
   - TTL: 5–30 минут.
   - Ограничение по объему (LRU), чтобы не раздувать память.

2. **L2 (дисковый, основной)**
   - Хранение уже ресайзнутых файлов.
   - Директория: например `var/cache/stock/`.
   - TTL: 7–30 дней (настраиваемо).
   - Очистка по TTL + ограничение максимального размера директории.

### Ключ кеша (обязательно полный)

Ключ должен включать:

- провайдер,
- исходный ID изображения у провайдера,
- `width`, `height`,
- `fit`,
- `format`,
- `quality`,
- версию алгоритма обработки (например `v1`).

Пример ключа:

`pexels:123456:w800:h450:fit-cover:webp:q82:v1`

Хеш ключа (SHA-1/SHA-256) использовать как имя файла на диске.

### Как уменьшить обращения к фотостоку

1. Сначала кешировать **результат поиска** (`query -> список candidate image ids`) на 15–60 минут.
2. Затем кешировать **итоговый ресайзнутый бинарник** (сам файл) на 7–30 дней.
3. Для одинаковых запросов на этапе miss применять **single-flight** (дедупликация параллельных запросов), чтобы 10 одновременных запросов не запускали 10 одинаковых загрузок/ресайзов.

### HTTP cache headers для клиентов/CDN

- `Cache-Control: public, max-age=86400, s-maxage=604800, stale-while-revalidate=86400`
- `ETag` на основе содержимого кеш-файла или ключа.
- Поддержать `If-None-Match` -> `304 Not Modified`.

## 4) Поток обработки запроса

1. Валидация `width/height/query/format`.
2. Нормализация запроса и сбор `cacheKey`.
3. Проверка L1 -> при hit сразу отдать.
4. Проверка L2 (диск) -> при hit загрузить в L1 и отдать.
5. При miss:
   - поиск в провайдере (`provider=auto`: сначала Pexels, потом Pixabay),
   - выбор кандидата (детерминированно с `seed`, либо первый релевантный),
   - загрузка оригинала/ближайшего размера,
   - ресайз в нужный размер,
   - запись в L2, прогрев L1,
   - отдача клиенту.

## 5) Ресайз и формат

- Для Node.js удобно использовать `sharp`.
- Дефолт формат: `webp` (лучший баланс размер/качество).
- Для превью и совместимости оставить `jpeg` fallback.
- Лимиты из конфига:
  - `STOCK_MAX_WIDTH`
  - `STOCK_MAX_HEIGHT`
  - `STOCK_CACHE_TTL_SEC`
  - `STOCK_CACHE_DIR`
  - `STOCK_CACHE_MAX_BYTES`

## 6) Ошибки и fallback

- Нет результатов по query -> `404` с JSON-ошибкой (`code: STOCK_NOT_FOUND`).
- Ошибка провайдера:
  - при `provider=auto` пробуем следующий;
  - если все упали -> `502` (`code: STOCK_PROVIDER_ERROR`).
- Невалидные размеры -> `400`.

## 7) Что сделать в коде (по шагам)

1. `packages/stock-provider` (или внутри `apps/server/lib/stock`):
   - клиенты Pexels/Pixabay;
   - унификация формата ответа (`id`, `author`, `url`, `thumbUrl`).
2. `packages/image-cache`:
   - L1 LRU + L2 disk store + single-flight lock.
3. Роут `apps/server/routes/stock/{width}/{height}/{query}.js` (через `express-openapi`).
4. OpenAPI описание + примеры ответов.
5. UI-конструктор вкладки stock (по аналогии с WxH/Avatars).

## 8) Риски и замечания

- Нужно проверить актуальные ToS каждого провайдера перед продакшеном.
- Кеш на локальном диске в Docker-контейнере лучше выносить в volume.
- Для высокой нагрузки на нескольких инстансах стоит перейти на общий кеш (S3/MinIO + Redis индекс).
